Безпека

Кожен запит захищений

Bearer токени зі збереженням у PostgreSQL і авто-ротацією, bcrypt (cost=12), HTTPS скрізь, ролева модель доступу на чотири рівні.

  • Bearer-токен у DB зі строком дії та авто-ротацією через X-Refresh-Token
  • bcrypt (cost=12) для хешування паролів
  • PostgreSQL CHECK та FOREIGN KEY constraints
  • HTTPS скрізь — Render забезпечує TLS
  • Ролі: soldier · operator · admin · platform_admin

Мультиплатформенність

Єдиний REST API обслуговує PWA, Android-додаток та будь-який HTTP-клієнт.

Service Worker, offline-режим, installable на домашній екран. Vanilla JS, без фреймворків.

Android

Kotlin + Retrofit2 + OkHttp3. Bearer Auth Interceptor, Coroutines для асинхронних запитів.

curl, Postman, будь-який HTTP-клієнт. Повна документація всіх 40+ ендпоінтів.

Документація

Склад програмного комплексу

Army Bank — це не лише «API + мобільний вигляд», а чотири зчеплені контури: бойовий банк для користувача, службові ролі в тому ж домені, окрема адмін-панель і маркетинговий вхід. Нижче — стислий зміст внутрішньої архітектурно-технічної документації (версія 1.5, 2026).

Backend + API

Python, Flask, модульні blueprint: auth, облік і транзакції, feature (донати, цілі, контакти, шаблони), admin, operator, platform, push. Єдиний JSON API з уніфікованими помилками та аудитом значущих дій.

  • PostgreSQL у проді; SQLite — для локального/демо режиму
  • schema.sql / schema_pg.sql, ініціалізація при старті
  • Пул з’єднань, gzip (Flask-Compress), batch GET /api/dashboard

PWA клієнт

index.html + vanilla JS: кабінет, операції, виплати, донати, накопичення, родина, аналітика, профіль. manifest.json (standalone, uk), Service Worker, Web Push, BroadcastChannel для синхрону вкладок.

  • Окремі оболонки admin.html, operator.html, platform.html під ролі
  • Один бекенд — усі клієнти ділять ту саму БД

Army Admin

Окремий SPA (репозиторій / статичний хост): вхід адміністратора, дашборд, користувачі з модалкою (інфо, дії, транзакції), виплати, аудит-логи. Працює через ті самі Bearer-сесії та /api/admin/*.

Публічний лендінг

Сторінка, на якій ви зараз: презентація продукту, стек, API-таблиці, посилання на live demo та GitHub. Точка маркетингового входу до бойового army-bank.onrender.com.

Інтеграційна модель: користувач переходить з лендінгу в PWA → авторизація → запити до Flask → репозиторії та PostgreSQL. Оператор і адміністратор заходять у свої інтерфейси, але дані й журнали спільні; критичні події фіксуються в транзакціях та audit_logs.

Функціонал

Що вже реалізовано в продукті

Перелік узгоджений із сценаріями з технічної документації та поточною кодовою базою: від реєстрації до платформеного адміністрування.

Користувач (soldier)

  • Реєстрація з валідацією ПІБ, телефону, email, пароля; автоматичне створення основного рахунку (номер на кшталт AB-100000 + id)
  • Вхід за identity (телефон або email) + пароль; сесійний Bearer-токен у PostgreSQL, TTL і оновлення сесії
  • Баланс, історія з фільтрами, поповнення, переказ за номером рахунку, експорт CSV
  • Бойові виплати (перегляд; демо/операторське нарахування)
  • Донати, цілі накопичення з внесками, довірені сімейні контакти
  • Шаблони платежів, сповіщення, аналітика, бюджетні ліміти в UI

Службові ролі

  • operator — нарахування виплат підрозділу, вузький контур /api/operator/*
  • admin — користувачі, ролі, системні транзакції, аудит, push-розсилка
  • platform_admin — платформені операції, демо-дані, bootstrap-ендпоінт для першого підняття ролі

Безпека та спостережуваність

  • X-Content-Type-Options: nosniff, CORS лише для дозволених origin
  • bcrypt для паролів; перевірка токена на кожному захищеному запиті
  • Журнал audit_logs для адміністративних дій

Сервер

Шар даних і конфігурація

database.py абстрагує PostgreSQL і SQLite через єдиний контекст з’єднання; репозиторії інкапсулюють SQL, сервіси — правила та транзакції, utils — валідація, токени, форматування.

Ключові таблиці

users (роль, ПІБ, телефон, email, статус), accounts (баланс, валюта, статус), sessions, transactions, payouts, donations, savings_goals, family_contacts, payment_templates, audit_logs, push-підписки та ін. Модель: один користувач — один основний рахунок, рух кошів узгоджується з журналом транзакцій.

Оточення (.env)

DATABASE_URL для PostgreSQL або шлях до SQLite; TOKEN_TTL_HOURS (типово 720); BASE_PATH для розгортання під префіксом (наприклад /bank); ADMIN_* для seed platform_admin; прапорець DEBUG.

Процеси

Життєвий цикл переказу

Спрощена послідовність того, що відбувається на сервері після POST /api/transactions/transfer (ізольована транзакція БД, перевірки цілісності).

01

Авторизація

Flask отримує Authorization: Bearer …, знаходить активну сесію в sessions, завантажує користувача та його основний рахунок.

02

Валідація

Перевірка суми, існування рахунку отримувача, заборона переказу «собі», достатність коштів на відправнику.

03

Оновлення балансів

У межах однієї DB-транзакції знімається сума з відправника, зараховується отримувачу; створюються симетричні записи в transactions (напрямки in / out).

04

Відповідь і клієнт

API повертає JSON з новими балансами; PWA оновлює стан, за потреби шле подію в BroadcastChannel, Web Push — отримувачу та відправнику.

DevOps

Розгортання

У репозиторії описані сценарії для Render: безкоштовний web-сервіс і опційно PostgreSQL через Blueprint (render.yaml). Збірка: pip install -r requirements.txt; старт: gunicorn -c deploy/gunicorn.conf.py backend.app:app.

Рекомендовано для продакшну

Постійна БД через DATABASE_URL (PostgreSQL на Render або власний інстанс). TLS забезпечує хостинг; для домену за nginx — проксування на Gunicorn з передачею X-Forwarded-*.

Локально та демо

SQLite у файлі (шлях у конфігу) для швидкого старту. Free tier Render «засинає» — перший запит після паузи може тривати десятки секунд; PWA може робити ранній ping /health при наявній сесії.

Точки входу: бойовий банк — army-bank.onrender.com; адміністративний UI — окремий хост (наприклад munister.com.ua/army-admin/); цей лендінг — маркетинговий шлях /army-bank/.

Сценарії

Практичне використання платформи

Окрім демонстрації інтерфейсу, Army Bank покриває операційні сценарії для кількох ролей. Це дозволяє команді використовувати один продукт як для щоденного банкінгу бійця, так і для службових задач підрозділу.

Боєць (soldier)

  • Швидкі перекази родині з історією та фільтрами
  • Контроль витрат і планування бюджету
  • Накопичення на особисті цілі та спорядження
  • Push-сповіщення про ключові фінансові події

Оператор (operator)

  • Нарахування виплат у службовому контурі
  • Окремі маршрути доступу через /api/operator/*
  • Контроль цільових операцій без доступу до адмін-функцій
  • Прозора взаємодія з єдиною транзакційною БД

Адміністратор (admin)

  • Керування ролями й життєвим циклом користувачів
  • Операційна аналітика, аудит-логи, контроль ризиків
  • Push-розсилки для критичних інформувань
  • Моніторинг платіжних процесів і модерація дій

Платформений контур

  • Єдина кодова база для PWA, admin і operator UI
  • Документований API для зовнішніх інтеграцій
  • Розгортання через Render + CI/CD workflow
  • Готова основа для масштабування функціоналу
KPI-рівень продукту: стабільний доступ до основних операцій, швидка адаптація нових користувачів, скорочення ручних дій операторів, прозорість адміністративного контролю та передбачувана підтримка через API-документацію.

Впровадження

План запуску та супроводу

Для команд, які хочуть запускати Army Bank як робочий сервіс, нижче наведено практичний план впровадження: від первинної інсталяції до підтримки в бойовому режимі.

01

Інфраструктура

Підняти середовище з PostgreSQL, налаштувати DATABASE_URL, SECRET_KEY, BASE_PATH, перевірити старт через /health та рольову модель доступів.

02

Інтеграція клієнтів

Підключити PWA або Android-клієнт до /api/auth/login та базових маршрутів /api/accounts/main, /api/transactions/*, включити обробку Bearer-токена та оновлення сесії.

03

Безпека та ролі

Розвести доступи для soldier/operator/admin/platform_admin, перевірити аудит дій і критичні сценарії операцій із балансом.

04

Експлуатація

Увімкнути CI smoke-перевірки, моніторити /api/admin/stats, контролювати Push-підписки та періодично перевіряти кеш-оновлення PWA після деплоїв.

Що отримує команда

  • Швидкий старт із готовою архітектурою та документацією
  • Один продукт для користувацького та службового контурів
  • Модульна база для подальшого розвитку функцій

Наступні кроки розвитку

  • Підключення додаткових каналів сповіщень
  • Розширення аналітики та anti-fraud правил
  • Поглиблення інтеграцій із зовнішніми системами

Часті питання

Додай Retrofit2 та OkHttp3 у build.gradle, створи ApiService з анотаціями @GET/@POST, налаштуй AuthInterceptor для Bearer токена. Базовий URL: https://army-bank.onrender.com/api/.

PWA отримує VAPID ключ через GET /api/push/vapid-public-key, підписується через PushManager.subscribe(), відправляє PushSubscription на POST /api/push/subscribe. Після транзакції — автоматичний push через pywebpush.

Кожна балансова операція — PostgreSQL-транзакція з SERIALIZABLE ізоляцією. Race condition виключений. Перед переказом: перевірка балансу, існування одержувача, валідності токена.

Ні. Замість JWT використовується Bearer-токен, згенерований через secrets.token_urlsafe(32) і збережений у таблиці sessions PostgreSQL разом з терміном дії. При кожному запиті сервер валідує токен в БД. Якщо до закінчення сесії залишається менше 7 днів — автоматично видається новий токен через заголовок X-Refresh-Token, клієнт оновлює його без перезаходу.

Render free tier "засинає" після 15 хвилин неактивності. При першому зверненні сервер "прокидається" — це займає 30–60 сек. Щоб мінімізувати цей ефект, PWA автоматично надсилає ping-запит на /health одразу при відкритті сторінки, якщо є збережена сесія, — ще до того, як користувач виконає першу дію.

Платформа має чотири рівні доступу: soldier — звичайний клієнт, бачить свій рахунок і транзакції; operator — нараховує виплати своїм підрозділам; admin — керує всіма юзерами, може коригувати баланс і робити push-розсилку; platform_admin — повний доступ до всіх адмін-панелей і статистики платформи.

Service Worker використовує стратегію stale-while-revalidate для CSS/JS (миттєве завантаження з кешу + фонове оновлення), cache-first для шрифтів і network-first для HTML-навігації. При відсутності мережі застосунок відкривається з кешу — видно останній стан балансу та транзакцій. Версія кешу змінюється при деплої, щоб клієнти підтягували свіжі файли.

Декілька рівнів оптимізації: CSS завантажується асинхронно (non-blocking), критичні стилі вбудовані inline; шрифти — обмежений набір ваг; Flask-Compress стискає відповіді gzip; пул з'єднань PostgreSQL зменшує накладні витрати на кожен запит; замість кількох паралельних викликів при відкритті кабінету — один batch-ендпоінт GET /api/dashboard.

Так. На Android — через банер "Додати на головний екран" або меню браузера. На iOS — Safari → "Поділитися" → "На екран «Додому»". Після встановлення застосунок запускається як нативний: без адресного рядка, із splash-screen і підтримкою push-сповіщень. Web App Manifest налаштований з іконками 72–512px та display: standalone.

Через Web BroadcastChannel. Коли одна вкладка виконує операцію (переказ, донат, видалення), вона публікує подію DATA_UPDATED — всі інші вкладки автоматично викликають refreshAllData() і відображають актуальні дані без перезавантаження сторінки.

Після входу викличте GET /api/transactions/export з заголовком Authorization: Bearer …. Параметри from та to (дати) за потреби обмежують період. У відповіді — CSV з BOM для коректного відкриття в Excel.

Сума має бути додатною й не перевищувати верхню межу, яку перевіряє бекенд (порядку 99 999 999,99 ₴ на одну операцію). У PWA можна додатково задати місячні бюджетні ліміти за типами витрат — вони допомагають контролювати витрати без зміни правил на сервері.

Для продакшн-деплою на Render: https://army-bank.onrender.com/api/. Спочатку POST /api/auth/login (тіло з телефоном/email і паролем), далі всі захищені маршрути — з Authorization: Bearer <token>. Якщо застосунок працює під префіксом шляху, додайте його перед /api.
