База даних

Схема PostgreSQL

Три основні таблиці з FK-зв'язками. Усі балансові операції — в межах транзакцій із SERIALIZABLE ізоляцією.

users
PK id SERIAL
UQ email VARCHAR(255) NOT NULL
password_hash VARCHAR(255) NOT NULL
full_name VARCHAR(255)
balance NUMERIC(12,2) DEFAULT 0
role VARCHAR(20) DEFAULT 'client'
created_at TIMESTAMP DEFAULT NOW()
transactions
PK id SERIAL
FK from_user_id INT → users.id
FK to_user_id INT → users.id
amount NUMERIC(12,2) NOT NULL
IDX type VARCHAR(20) — transfer/deposit
description TEXT
IDX created_at TIMESTAMP DEFAULT NOW()
push_subscriptions
PK id SERIAL
FK user_id INT → users.id
endpoint TEXT NOT NULL
p256dh_key TEXT
auth_key TEXT
created_at TIMESTAMP DEFAULT NOW()
users 1 ──── ∞ transactions через from_user_id та to_user_id
users 1 ──── ∞ push_subscriptions один юзер може мати підписки на кількох пристроях

Процес

Від реєстрації до переказу

1

Реєстрація через API або PWA

POST /api/auth/register — email + пароль. Сервер хешує пароль через bcrypt (cost=12), створює запис у таблиці users з балансом 0₴. Відповідь — JWT токен.

↓
2

Авторизація — Bearer токен

POST /api/auth/login верифікує пароль через bcrypt.checkpw(). При успіху — jwt.encode() з user_id та exp. Клієнт зберігає токен і додає Authorization: Bearer … до кожного запиту.

↓
3

Переказ коштів — атомарна операція

POST /api/transactions/transfer відкриває транзакцію PostgreSQL: перевіряє баланс відправника, вираховує суму, зараховує одержувачу, записує запис у таблицю transactions. Або все — або нічого. Після успіху — Web Push обом сторонам.

FAQ

Часті питання

Додай Retrofit2 та OkHttp3 у build.gradle, створи ApiService інтерфейс з анотаціями @GET/@POST, налаштуй AuthInterceptor для додавання Bearer токена. Data classes для всіх відповідей описані у репозиторії. Базовий URL: https://army-bank.onrender.com/api/.

PWA отримує VAPID публічний ключ через GET /api/push/vapid-public-key, підписується через браузерний PushManager.subscribe(), відправляє PushSubscription на POST /api/push/subscribe. Після кожної транзакції сервер надсилає Push через pywebpush — сповіщення приходить навіть із закритою вкладкою.

Кожна балансова операція виконується у PostgreSQL-транзакції з SERIALIZABLE рівнем ізоляції. Це виключає race condition навіть при одночасних запитах. Перед переказом сервер перевіряє: чи є баланс, чи існує одержувач, чи валідний токен — і лише тоді виконує UPDATE у БД.

Admin має доступ до /api/admin/* ендпоінтів: перегляд усіх юзерів, ручне коригування балансів, перегляд усіх транзакцій, Push-розсилка. Flask middleware перевіряє role == 'admin' у JWT payload перед кожним admin-запитом.
